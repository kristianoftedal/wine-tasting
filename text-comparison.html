① Token Overlap

Jaccard Similarity

Measures the ratio of shared tokens to total unique tokens between two texts.

J(A, B) = |A ∩ B| / |A ∪ B|
Text A: "machine learning is fun"
Text B: "learning machine is powerful"

Intersection = {machine, learning, is} = 3
Union = 5 unique tokens

J = 3/5 = 0.60

Good for

  • Duplicate detection
  • Tag and keyword matching
  • Search index deduplication

Key weakness

"car" and "automobile" score 0. No understanding of meaning — only of token identity.

① Token Overlap

Dice Coefficient & Overlap Coefficient

Dice — weights intersection more

Dice(A,B) = 2|A ∩ B| / (|A| + |B|)
A = {machine, learning, is, fun}
B = {learning, machine, is, powerful}

Dice = 2×3 / (4+4) = 0.75
Jaccard was 0.60 — Dice always ≥ Jaccard

Overlap — detects containment

Overlap(A,B) = |A ∩ B| / min(|A|, |B|)
A = {cat, sat, on} (3 tokens)
B = {the, cat, sat, on, mat} (5 tokens)

Overlap = 3 / 3 = 1.0
← A is fully contained in B

Used in ontology matching and query-in-document detection.

② Edit Distance

Levenshtein Distance

Minimum number of insertions, deletions, and substitutions to transform one string into another.

"kitten" → "sitting"

k → s (substitution)
e → i (substitution)
+ g (insertion)

Distance = 3
Similarity = 1 − 3/7 ≈ 0.57

Damerau-Levenshtein

Adds transposition — better models real typing errors.

"hte" → "the"
Levenshtein = 2
Damerau = 1 (swap h and t)

Good for

  • Spell checking and autocorrect
  • OCR post-processing
  • Fuzzy product-code matching
② Edit Distance

Jaro-Winkler Similarity

Optimised for short strings like names. Rewards matching characters in proximity and boosts a shared prefix.

Jaro = ⅓ · (m/|s₁| + m/|s₂| + (m−t/2)/m)
m = matching chars · t = transpositions/2
Jaro-Winkler = Jaro + p·ℓ·(1−Jaro)
ℓ = shared prefix (max 4) · p = 0.1
"MARTHA" vs "MARHTA"
Jaro ≈ 0.944
Jaro-W ≈ 0.961 ← prefix "MAR" bonus

Good for

  • Person and company name matching
  • Record linkage across databases
  • Short identifier deduplication
② Edit Distance

Hamming Distance & LCS

Hamming — positional differences

Counts positions where two equal-length strings differ. Used in error-correcting codes.

1011101
1001001
x x distance = 2

Longest Common Subsequence

Longest ordered token sequence shared by both texts — gaps allowed. Foundation of git diff.

A: "the cat sat on the mat"
B: "the dog sat on a hat"

LCS: "the sat on" (3 tokens)
Similarity = 2×3 / (6+6) = 0.50
  • Diff and merge tools
  • Plagiarism detection
  • DNA sequence comparison
③ Sub-Word Patterns

N-grams, MinHash & SimHash

N-grams / Shingling

Break text into overlapping windows of n characters or words, then compare the sets. Tolerant of typos and minor reorderings.

char bigrams("hello"):
he · el · ll · lo
  • Typo-tolerant search
  • Language detection
  • Plagiarism checking

MinHash

Approximates Jaccard using random hash functions. Compresses a document to a compact signature — no need to compare full token sets.

P(minhash match) = Jaccard similarity

  • Web-scale near-duplicate detection
  • Locality-Sensitive Hashing

SimHash

Maps text to a fixed-length bit vector. Hamming distance between SimHashes estimates document similarity in O(1). Used by Google's crawl dedup pipeline.

doc_A: 10110011
doc_B: 10110001
diff: xx → dist = 2
③ Sub-Word Patterns

Phonetic Similarity & Sequence Alignment

Phonetic — compare by sound

Converts words to pronunciation codes, then compares codes. Spelling differences that don't affect sound are ignored.

Soundex("Smith") = S530
Soundex("Smyth") = S530 ← match
Soundex("Smythe") = S530 ← match
  • Healthcare patient record linkage
  • Name search across transliterations
  • Speech-to-text post-correction

Sequence Alignment — borrowed from bioinformatics

Find the optimal alignment with configurable gap and mismatch penalties.

  • Needleman-Wunsch — global alignment end-to-end
  • Smith-Waterman — local alignment, best sub-region

Applied to DNA, proteins, and advanced fuzzy matching where gaps carry domain-specific costs.

④ Statistical Weighting

TF-IDF

The first step toward importance — not just presence. Rare, specific words carry more signal than common ones.

TF-IDF(t, d) = count(t)/|d|  ×  log(N / df(t))

Intuition

  • "the", "is", "and" → weight ≈ 0
  • "transformer", "backpropagation" → high weight
  • A word rare across the corpus is diagnostic

Usage

Convert each document to a TF-IDF vector — one dimension per vocabulary term. Compare documents using cosine similarity on those vectors.

Standard for information retrieval since the 1970s. Sparse, fast, no training required.

④ Statistical Weighting

BM25 — Okapi Best Match 25

The successor to TF-IDF. Default ranker in Elasticsearch, Solr, Lucene. Fixes two known failure modes.

TF saturation

In TF-IDF, a word appearing 100× scores 10× better than 10×. In BM25, returns diminish — keyword stuffing no longer inflates scores.

Length normalisation

A short, precise document isn't penalised for having lower raw term counts than a longer one.

Where BM25 dominates

  • Keyword search over large corpora
  • First-stage candidate recall in RAG pipelines
  • Legal and academic document retrieval

BM25 retrieves the top-1 000 candidates cheaply. Embeddings then rerank the top-100 accurately. This two-stage pattern is the industry standard.

④ Statistical Weighting

Cosine Similarity

Represents documents as vectors and measures the angle between them — not their distance. Two documents with the same word distribution but different lengths score 1.0.

cos(θ) = (A · B) / (‖A‖ · ‖B‖)
Angle θCosineMeaning
0°1.0Identical direction
45°0.71Similar
90°0.0Orthogonal — unrelated
180°−1.0Opposite

Works identically whether vectors are TF-IDF sparse or dense neural embeddings.

This raises a question

Why does representing text as a point in geometric space and measuring an angle give us meaningful similarity?

It works. But why is geometry the right language for this problem at all?

Historical Parallel · Part 1

Einstein Had the Physics — But Not the Language

By 1907, Einstein had a clear physical intuition:

Gravity is not a force acting through space. It is the curvature of spacetime itself. A planet orbiting the sun is simply moving in a straight line through curved geometry.

The physics was clear. The problem was writing it down.

The obstacle

Newton's equations are written for flat, 3D space with a fixed coordinate system. Einstein needed laws that worked in any coordinate system — flat, curved, accelerating.

Classical math had no way to express this. He was stuck for five years.

The question

What mathematical language lets you describe a law that holds regardless of how you choose to describe space?

Historical Parallel · Part 2

A Physical Problem Was a Geometric Problem All Along

In 1912, Einstein's mathematician friend Marcel Grossmann showed him Bernhard Riemann's 1854 work on curved geometry, and the tensor calculus of Ricci and Levi-Civita.

Tensors are coordinate-independent by construction — exactly the language Einstein needed. Within three years he had the field equations:

Gμν = (8πG / c⁴) · Tμν

Curvature of spacetime = energy–momentum of matter

What changed

A problem stated in the language of physics — forces, masses, trajectories — turned out to be a problem about the geometry of a manifold.

Once expressed geometrically, the solution was inevitable. The right abstraction unlocked everything.

The lesson

Some problems look hard in their native domain, but become tractable — even elegant — when reframed geometrically.

The Parallel

NLP Made the Same Move

Measuring meaning similarity is hard in the language domain. It becomes tractable when reframed as a geometric problem.

Einstein's reframing

Physics→Geometry
Gravity→Curvature of spacetime
Mass / Energy→Metric tensor gμν
Trajectory→Geodesic path
Proximity→Spacetime interval ds²

NLP's reframing

Language→Geometry
Word / sentence→Point in vector space
Meaning→Position of the vector
Semantic similarity→Angle between vectors
Analogy (king → queen)→Translation in space
The Parallel

Why the Geometric Approach Wins

The reframing unlocks capabilities that are impossible in the original token-matching approach.

Synonyms are free

Words with similar meaning land near each other in space — no thesaurus or synonym list required. The geometry learns the relationships from text.

car ≈ automobile ≈ vehicle
(nearby vectors, no rules)

Language-independent

The same geometric position can be reached from different surface forms — English, Norwegian, slang, formal register. Meaning transcends spelling.

"bær" ≈ "berries" ≈ "bærete"
(close vectors, different words)

Composition is natural

Multiple descriptors combine in vector space. A phrase embedding is a point that captures the joint meaning of its parts — not just their intersection.

Structure emerges

Relations become arithmetic: king − man + woman ≈ queen. The geometry captures structure that was never explicitly programmed.

Computationally efficient

Once text is a vector, comparison is a dot product — fast on modern hardware. Billions of comparisons per second via approximate nearest-neighbour (ANN) search.

Graceful degradation

No shared words needed — even two completely different wordings of the same idea produce nearby vectors. Token-based methods would score zero.

④ Cosine in Vector Space

Seeing the Angle

The key insight

Two identical documents of different lengths produce parallel vectors. Their distance differs — but their angle is 0°, so cosine = 1.0. Direction encodes topic; magnitude encodes length.

In 1 536 dimensions

OpenAI's embedding model produces vectors with 1 536 dimensions. We cannot visualise them — but the same cosine formula applies exactly. Small angle = similar meaning.

Scale

  • cos(θ) = 1 → identical meaning
  • cos(θ) = 0 → completely unrelated
  • cos(θ) = −1 → opposite meaning
⑤ Semantic Similarity

Word Embeddings

Each word becomes a dense vector in a learned space where distance encodes meaning. Trained on billions of tokens — the geometry emerges from context patterns.

vector("king") − vector("man")
+ vector("woman")
≈ vector("queen")

vector("Paris") − vector("France")
+ vector("Germany")
≈ vector("Berlin")

Semantic relations are geometric translations — never explicitly programmed.

Key architectures

  • Word2Vec — skip-gram / CBOW (Google, 2013)
  • GloVe — co-occurrence matrix factorisation
  • FastText — subword n-grams, handles OOV words

Key limitation

One fixed vector per word regardless of context. "Apple" has the same vector for fruit and for the tech company. Transformers solved this.

⑤ Semantic Similarity

Word Mover's Distance

How much geometric "work" is needed to move all words in one document to match the distribution of another? Based on optimal transport theory.

D1: "Obama speaks to the media"
D2: "The president greets the press"

obama → president (short hop)
speaks → greets (short hop)
media → press (short hop)

WMD = low — synonymous, similar meaning

Why it's powerful

No shared words needed. Two documents with zero lexical overlap can score very similar — handled by routing word vectors through space optimally.

Tradeoffs

  • No lexical overlap needed
  • Expensive: O(p³ log p) — slow on long documents
  • Requires pre-trained word vectors
⑤ Semantic Similarity

Sentence & Document Embeddings

Encode an entire sentence into a single dense vector — the state of the art for most similarity tasks.

"I love programming"
"Coding is enjoyable"
→ nearby vectors, high cosine

"I love programming"
"The weather is cold"
→ distant vectors, low cosine

Key models

  • Sentence-BERT — bi-encoder, fast inference
  • Universal Sentence Encoder — multilingual
  • OpenAI text-embedding-3-small/large — 1 536 / 3 072 dims
  • E5, BGE, GTE — open-source SOTA

Approach

  1. Send text to embedding model
  2. Compute cosine similarity
  3. Scale to 0–100 for display
⑤ Transformer Models

Transformer-Based Similarity

Transformers produce contextual representations — the same word gets a different vector in different sentences.

"Apple released a phone"
→ Apple ≈ [tech, company, iOS …]

"Apple pie tastes good"
→ Apple ≈ [food, fruit, sweet …]

Bi-encoder vs Cross-encoder

  • Bi-encoder — encode each text independently; fast; used for ANN search
  • Cross-encoder — encode the pair together; sees interaction; much more accurate; used for reranking

Key models

  • BERT / RoBERTa — masked language model
  • MiniLM — lightweight, fast bi-encoder
  • ColBERT — late interaction
  • GPT / Claude — generative models as encoders

Understands

  • Context and grammatical nuance
  • Ambiguity and polysemy
  • Cross-lingual semantic equivalence

A Real-World Application

A Norwegian wine tasting app — comparing user notes to expert descriptions using a hybrid NLP pipeline

Application

The Problem

Users taste wines at events and write free-text notes. The system scores how closely their description matches the official producer description.

User writes
"Bærete med mineralaktig ettersmak, tanninrik og balansert"
Official description
"Bær og mineral. Rik på tannin med god frukt og lang avslutning"
Very few shared tokens — but clearly describing the same wine

Why pure semantic similarity isn't enough

Embeddings give a high score to anything that's roughly about wine. But the goal is also to reward users who use specific, precise wine vocabulary — "solbær" over "frukt", "grafitt" over "mineral".

Why pure token matching isn't enough

Norwegian wine notes use many morphological variants — "bærete", "mineralaktig", "eikepreget" — that share zero tokens with the canonical forms in the official description.

Clean & normalise

Remove noise and standardise Norwegian wine vocabulary. "Bærete", "bærfrukt", "bærrik" all map to the canonical form "bær" before any scoring.

Vocabulary match

Compare against ~200 expert wine descriptors. Each match is weighted by rarity — uncommon, precise terms earn more credit than generic ones.

Semantic embed

Encode the cleaned text as a 1 536-dim vector using an embedding model. Measure cosine similarity. Captures meaning even when words don't overlap.

Application

Two Complementary Scoring Tracks

Track A — Semantic

Encodes the cleaned note as a high-dimensional vector and measures the cosine angle against the official description's vector.

Rewards: overall meaning, paraphrasing, capturing the spirit of the wine even with different words.

base score = cosine(embed(userNote),
embed(officialDesc))
× 100

Track B — Lexical precision

Counts how many expert wine descriptors appear in both texts — weighted by how rare and specific each term is across the wine corpus.

Rewards: using precise vocabulary — "solbær" over "frukt", "grafitt" over "mineral".

precision = IDF-weighted overlap
of matched wine terms

The tracks are complementary. Semantic alone rewards vague notes. Lexical alone misses paraphrases. Together they reward notes that are both meaningful and precise.

Application

The Scoring Formula

score = min(100,  semanticScore + 0.35  ×  precision)
precision = (vocabularyOverlap + categoryOverlap) / 2

semanticScore (0–100)

Cosine similarity of embedding vectors. Captures overall meaning — paraphrases, synonyms, different languages.

vocabularyOverlap

IDF-weighted Jaccard on matched wine descriptors. Rare terms score higher than generic ones.

categoryOverlap

Partial credit for matching the right flavour category even without the exact term — "bær" vs "kirsebær" both belong to Frukt/bær.

Vocabulary precision adds up to +35 points on top of the semantic base — rewarding specific vocabulary while the embedding base ensures paraphrases still score well.
